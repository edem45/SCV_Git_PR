# Работа с GIT и GITHUB

## 1. Проверка наличия установленного GIT
В терминале выполнить команду `git version`

Если Git установлен появится сообщениие с информацией о версии программы иначе будет сообщение об ошибке.

## 2. Установка Git
Загружаем последнюю версию Git с сайта https://git-scm.com/download.

Устанавливаем с настройками по умолчанию.

## 3. Настройка Git
При первом использовании Git на необходимо представиться. Для этого нелбходимо ввести в терминале две команды:

 ```git config --global user.name «Ваше имя английскими буквами» ```

```git config --global user.email ваша почта@example.com ```

## 4. Иициализация репозитория 
Получить репозиторий можно двумя способами:
1. В терминале перейти к папке в которой хотим создать репозиторий. Выполняем команду:
```
git init
```
В исходной папке появится скрытая папка **.git**

2. Клонировать существующий репозиторий Git с любого места. Сделать можно это с помощью команды:
```
git clone <адрес или ссылка>
```
## 5. Добавление файлов в область подготовленных и апись изменений в репозиторий 

Для того чтобы сохранить изменения в репозитории неоюходимо выполнить следующие пункты:

|1.Команда *CTRL+S*|2.Команда *git add <имя файла>*|3.Команда *git commit -m <Примечание>*|4.Команда *git commit -a -m <Примечание>*|
|:---:|:---:|:---:|:---:|
|После каждого изменения файла необходимо сохранить его использовав комбинацию *CTRL+S* либо настроить *АВТОСОХРАНЕНИЕ*|Добавить отдельный файл в область подготовленных файлов можно параметром add с указанием имени файла|При создании коммита в репозитории можно добавить однострочное сообщение с помощью параметра commit с флагом -m. Само сообщение вводится непосредственно после флага, в кавычках.|Также можно использовать комбинированную команду *git commit -a -m <Примечание>* которая в свою очередь добавляет файл и фиксирует изменения|


## 6. Проверка статуса репозитория
Просмотреть статус нужного репозитория можно по ключевому слову status: его действие распространяется на подготовленные, неподготовленные и неотслеживаемые файлы.
```
git status
```
## 7. Просмотр истории коммитов с изменениями

Просматривать изменения можно с помощью команды:
```
git log
```
Он отображает список последних коммитов в порядке выполнения.
Кроме того:
> Добавив **--oneline** вы можете посмотреть сокращенный вывод содержащий одну строку с сокращенным хешом.

> Добавив флаг **-p** вы можете подробно изучить изменения, внесённые в каждый файл.

## 8. Просмотр заданного коммита 

Просмотреть полный список изменений, внесённых конкретным коммитом, можно с помощью параметра show, указав идентификатор или хеш коммита. Значение хеша уникально для каждого коммита, созданного в вашем репозитории.
```
git show <хеш>
```
либо
```
git checkout <хеш>
```

## 9. Возврат к актуальному состоянию и продолжение работы

Необходимо ввести след команду:
***
```
git checkout master
```

## 10. Просмотр изменений до коммита 

Необходимо ввести след команду:
```
git diff
```
Для просмотра подготовленных изменений необходимо добавить флаг --staged
```
git diff --stages
```

Также можно указать имя файла как параметр и просмотреть изменения, внесённые только в этот файл.
```
git diff <file name>
```

## 13. Игнорирование файлов

Для того чтобы исключить из отслеживания в репозитории определенных файлов и папки необходимо создать файл ***.gitignore*** и записать в него их названия или шаблоны соответствующие таким файлам или папкам.

Например, если у нас не один 1 а 20, переписывание имен файлов займет время, поэтому целесообразней пользоваться шаблонами файлов,как показанов таблице:

|Шаблон|Примеры соответствия|Пояснение*|
|:---:|:---:|:---:|
|**/logs|logs/debug.log logs/monday/foo.bar build/logs/debug.log|Добавьте в начало шаблона две звездочки, чтобы сопоставлять каталоги в любом месте репозитория.|
|**/logs/debug.log|logs/debug.log build/logs/debug.log но не logs/build/debug.log|Две звездочки можно также использовать для сопоставления файлов на основе их имени и имени родительского каталога.|
|*.log|debug.log foo.log .log logs/debug.log|Одна звездочка — это подстановочный знак, который может соответствовать как нескольким символам, так и ни одному.|
|debug.log|debug.log logs/debug.log|По умолчанию шаблоны соответствуют файлам, находящимся в любом каталоге|

## 14. Создание веток

По умолчанию имя основной ветки в Git - *Master*.

1. Создать ветку можно командой :
```
git branch <branch_name>
```
2. И также мы получим тот же результат при использоqвании git checkout с ключом -b:
```
git checkout -b <name of branch>
```
Если пользователю нужно получить список определенного множества веток, то тогда можно воспользоваться ключами. Одними из самых распространенных будут:

* r — при использовании этого ключа мы получим список удаленных веток,
* a — используя этот параметр, в выводе будут удаленные и локальные ветки.

Список веток в репозитории можно посмотреть с помощью команды  `git branch` .
Текущая ветка будет отмечена звездочкой:
*\*master*

## 15. Слияние веток и разрешение конфликтов
Для слияния выбранной ветки и текущей нужно выполнить команду:
```
git merge <Branche_name>
```
Если была изменннннена одна и та же часть файла в обеих ветках, то может возникнуть конфликт, который потребует участия пользователя.
VSCode предлагает варианты разрешения.

После выбора варианта слияния (при разрешении конфликта) необходимо создать коммит `Разрешения конфликта`:
```
git commit -am "разрешение конфликта"
```

## 16. Удаление веток

Для того чтобы удалить ветку, мы не должны в ней находится, поэтому с помощью команды `git branch` смотрим на какой ветке мы находимся, при необходимости с помощью команды `git switch <branch_name>` перключаемся на другую ветку.

Непосредственно для удаления используем команду:
```
git branch -d <branch_name>
```
Но если ветка не полностью слита, программа выдаст ошибку, нужно либо слить ветку и повторить команду, либо если вы уверены что не боитесь потерять информацию которая находится в неслитой ветке, можно использовать команду принудительного удаления:
```
git branch -D <branch_name>
```
## 17. Работа с удаленными репозиториями

`Для начала работы с удаленными репозиториями на необходимо иметь аккаунт на GitHub(к примеру), так как кроме клонирования репозитория с сайты, мы будет загружать туда свой локальный репозиторий, либо *форкать* чей-либо: выгружать на свой локальный ПК, изменять и выгружать обратно предлагая свои изменения автору проекта делая **PULL REQUEST**`

### 1.Копирование репозитория с удаленного хранилища (к примеру GitHub)

1. Найти в хранилище интересующий нас репозиторий
2. Скопировать ссылку на него 
3. На нашей локальной машине создать пустую папку (! Не репозиторий), так как мы будет копировать в нее уже имеющийся репозиторий, если мы скопируем его в наш локальный репозиторий, то будет **Репозиторий > в Репозитории**.
4. Зхаодим в нашу пустую папку, в терминале вызываем команду:
```
git clone <url адрес репозитория>
```
5.Переходим в скопированную папку с помощью команды `cd _folder name_`, и приступаем к работе с проектом.

### 2.Создание репозитория на локальном ПК и выгрузка в удаленное хранилище

1. Создаем репозиторий на локальном ПК либо открываем уже имеющийся.
2. Свзываем локальный репозиторий с удаленным с помощью команды:
```
git remote add Короткое_имя <url адрес>
```
**Короткое_имя** указывается для того чтобы мы в последующий обращениях к URL-адрессу репозитория ссылались на него.

Просматривать удалённые URL-адреса можно параметром remote с флагом -v. Этот параметр отображает удалённые подключения к другим репозиториям:
```
git remote -v
```
Получить подробные сведения об удалённом репозитории можно с помощью параметра remote show с указанием имени репозитория — например, origin:
```
git remote show <короткое_имя>
```
Эта команда отображает список веток, связанных с удалённым репозиторием, а также рабочих станций, подключённых для получения и отправки файлов.

3. Отправлять изменения в удалённый репозиторий можно параметром push с указанием имени репозитория и ветки.
```
 git push <короткое_имя > <имя ветки>
```

Также Git не даст выгрузить репозиторий на удаленный серевер, если на нем не такой ветки, как в локальном, тогда нужно следовать указанием git и ввести комагду для создания ветки на удаленном репозитории с названием локальной:
```
git push -u <короткое_имя> <имя ветки>
```
### 3. Выгрузка с удаленного репозитория и слияния с локальным (составная команда)

Для загрузки изменений из удалённого репозитория используется параметр pull. Он скачивает копию текущей ветки с указанного удалённого репозитория и объединяет её с локальной копией:
```
git pull

```
### 4. Создание Fork и pull request

1. Для того, чтобы создать ответвление проекта, зайдите на страницу проекта и нажмите кнопку «Создать ответвление» («Fork»), которая расположена в правом верхнем углу.

2. Выгрузить в локальный репозиторий (Подзаголок 1 см.выше)

3. Перйти в наш репозиторий с помощью команды `cd <repozitory_name>`

4. Создайте новую ветку `git branch <branch_name>`

5. Внесите изменения и закомитте

6. Отправте изменения (Подзаголовок 2 пункт 3 см.выше )

7. Зайти на удаленный репозиторий и создать Pull Request(Гитхаб либо предложит сам при обновлении страницы, либ оперейти во вкладку Pull Request, будте внимательны при создании запроса на изменения, на необходимо указать имя новой ветки в нашем репозитории данного проекта, если мы останемся на главной ветки, то сайт изменений не найдет и предлагать будет нечего)
8. Ждем ответа от автора проекта.
